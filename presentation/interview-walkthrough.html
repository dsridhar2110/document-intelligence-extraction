Document Intelligence — I taught the pipeline to read documents so people don't have to

It takes messy financial documents — normal PDFs, scanned PDFs, even a web page — and turns each one into a clean row of data: company, period, revenue, profit. It only asks a human when it isn't sure. Built as the "automated sourcing and extraction" piece of a financial data platform.

01 · What this project is for

The problem, in one breath

A financial-analytics platform is only as good as the data feeding it — and a huge amount of that data arrives as documents: filings, annual reports, scanned statements. Someone has to read them and type the numbers in. That's slow, expensive, and it doesn't scale as the pile of documents grows. So I built a pipeline that reads the document instead — it pulls out the key figures, checks them, and hands back a tidy record. If it isn't confident, it flags the document for a person rather than guessing. It's a small, honest version of the sourcing-and-extraction capability at the heart of a data-management framework — exactly what the LSEG role is about.

02 · The whole thing, simply

What I built, step by step

1

I fetch the documents automatically.

A little scraper pulls the source files — company filings and reports — and logs where each one came from (URL, time). That's the sourcing / crawling part.

Why it matters: the pipeline starts itself; nobody hand-downloads files.

2

I turn the document into text — even the scanned ones.

Normal PDFs already contain text, so I read it directly. Scanned PDFs are just pictures, so I run OCR (optical character recognition) — software that "reads" the image and gives back the words. That's the hard bit most tools skip.

Why it matters: real document piles are always a mix of clean files and scans.

3

An LLM pulls out the fields — but grounded, not guessing.

I give a large language model a strict instruction: "from this text, return the company, period, revenue, net income, EPS and currency, as JSON." To keep it honest on long documents I use RAG — I first retrieve the relevant page, then ask, so it answers from the actual document rather than memory. The answer must fit a fixed schema (src/schema.py).

Why it matters: "structured outputs + RAG" is what turns an LLM from a chatbot into a reliable extractor.

4

I check the answer before I trust it.

Validation guardrails: revenue can't be negative, the currency must be a real code, the period must look like a period, and net income shouldn't exceed revenue. Anything that fails — or where the model's confidence is low — is routed to a human-review queue instead of flowing straight through.

Why it matters: in a regulated, financial setting, "flag it for a human" beats "guess confidently".

5

A small screen for the human reviewer.

A Streamlit table shows the flagged documents side by side with the extracted numbers, so a person confirms or corrects in seconds. Everything else is already done.

03 · If I can't share my screen

Said out loud

What model did you use, and why?

A hosted large language model, but with two seatbelts on it. First, RAG — I retrieve the right part of the document and feed only that in, so it answers from the real text. Second, structured outputs — the model has to return JSON that fits a fixed schema, and I validate that schema before I accept anything. I didn't train a model; extraction like this is better solved with good prompting plus guardrails, and it's far easier to explain and audit.

key terms: LLM · RAG (retrieval-augmented generation) · structured outputs · prompt engineering

How did you know it worked? (evaluation)

I hand-labelled a small gold set — the correct numbers for ~15–20 documents — and measured field-level accuracy against it. I also track how many documents go straight through versus how many get sent to a human. Those two numbers are the honest scorecard.

key terms: gold set · field-level accuracy / F1 · straight-through rate

What other techniques did you use?

OCR for the scanned files, a simple rule/regex baseline to compare the LLM against, and validation rules as post-processing. The baseline matters — it proves the LLM is actually earning its place.

key terms: OCR · baseline · post-processing / guardrails

04 · What every screen/chart means

Each view, two ways

The extraction table (document → fields)

In the interview: each row is one document turned into structured data — the numbers the model pulled out, with a confidence and a pass/flag status.

🔍 For my understanding: this is the whole point of the project made visible — a picture became a row of clean data. The status column is the trust: green means "checked and safe", amber means "a human should look". Why a status at all? because being right most of the time isn't enough in finance — you have to know which ones you're unsure about.

The accuracy scorecard (gold set)

In the interview: field-level accuracy against my hand-labelled answers, plus the straight-through rate.

🔍 For my understanding: accuracy is "of all the numbers it pulled, how many matched the truth I wrote down." Why field-level, not document-level? because getting 5 of 6 fields right on a document is useful — a whole-document pass/fail would hide that. And the straight-through rate is really a business number: how much manual typing did I actually remove?

07 · The conversation

They ask / I'd say

Why an LLM and not just regex or a template?

Regex works until the layout changes — a different filing format and it breaks. The LLM reads meaning, not position, so it survives messy, varied documents. But I keep the regex as a baseline to prove the LLM is worth its cost.

Land this: "meaning, not position." · For me: that's exactly why it generalises across document types.

What is RAG, really?

Retrieval-Augmented Generation. Instead of dumping a 200-page report into the model and hoping, I first fetch the few relevant chunks, then ask the model using only those. It grounds the answer in the real document and keeps it cheap.

For me: retrieve first, then generate — that's the whole idea.

How do you stop it hallucinating a number?

Three ways: RAG grounds it in the actual text; structured outputs force it into a fixed schema; and validation rules catch the impossible (negative revenue, profit bigger than revenue, a fake currency). Low confidence or a failed rule means a human looks — I never let a shaky number flow straight through.

Land this: "ground it, constrain it, check it, and when in doubt ask a human."

Why measure a "straight-through rate"?

Because that's the business value in one number: the share of documents that needed no human at all. Accuracy says it's right; straight-through says how much manual work it actually removed.

What would you do to make this production-grade?

Fine-tune an extractor on more labelled docs, add table-structure recognition for statements, monitor extraction accuracy over time (drift), and put the whole thing behind versioning and CI — the MLOps wrapper.

08 · The same idea at LSEG

How this bridges to the job

LSEG's whole business is turning the world's financial information into clean, reliable data. A data-management framework has to acquire, clean and transform content at scale — and a lot of that content is unstructured documents. This project is that exact first mile: source → extract → structure → validate, with a human in the loop where it matters.

The honest framing: this is a toy next to LSEG's real pipelines — a handful of documents, not millions. But it's the same shape, I measured it honestly, and it taught me where the hard parts really are: the scans, the validation, and knowing when not to trust the model.

09 · One thing worth knowing cold

What "structured / semi-structured / unstructured" means

Structured data already sits in neat rows and columns (a database table). Semi-structured has some tags but no fixed table (JSON, a form). Unstructured is free-form — the text of a PDF, a scanned page. This whole project is a machine for moving data from the unstructured end to the structured end. Say that sentence and you've said the point.
